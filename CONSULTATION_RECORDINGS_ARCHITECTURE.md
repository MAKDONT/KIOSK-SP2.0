# Consultation Recordings Architecture - Codebase Exploration

## 1. DATABASE SCHEMA FOR CONSULTATIONS

### Primary Table: `queue` (NOT a separate consultations table)
**Location:** [supabase-schema.sql](supabase-schema.sql#L81)

All consultation/session data is stored in the **queue** table with these key fields:

```sql
CREATE TABLE IF NOT EXISTS queue (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  student_id TEXT NOT NULL REFERENCES students(id),
  faculty_id TEXT NOT NULL REFERENCES faculty(id),
  status TEXT DEFAULT 'waiting',  -- waiting, next, serving, completed, cancelled
  queue_number TEXT,
  position INTEGER,
  entry_time TIMESTAMPTZ DEFAULT NOW(),
  start_time TIMESTAMPTZ,
  end_time TIMESTAMPTZ,
  meet_link TEXT,
  recording_enabled BOOLEAN DEFAULT FALSE,  -- ← Recording flag
  source TEXT,  -- kiosk, web
  purpose TEXT,
  queue_date DATE DEFAULT CURRENT_DATE,
  time_period TEXT,
  student_email TEXT,
  department_id BIGINT REFERENCES departments(id),
  college_id UUID REFERENCES colleges(id)
);
```

### Key Recording-Related Fields in queue Table:
- **`recording_enabled`** (BOOLEAN): Flag indicating if recording is enabled for this consultation
- **`id`**: Consultation ID (references recording via `consultation_id` metadata in filenames)
- **`faculty_id`**: Faculty member conducting consultation
- **`student_id`**: Student in consultation
- **`start_time`**: When consultation started
- **`end_time`**: When consultation ended

### Related Tables:
- **faculty**: `id, name, full_name, email, status`
- **students**: `id, student_number, full_name, email, course`
- **departments**: `id, name, code, college_id`
- **colleges**: `id, name, code`

## 2. RECORDING STORAGE & LINKAGE

### Supabase Storage Bucket Structure
- **Bucket Name**: `consultation-recordings` (default, configurable via `SUPABASE_RECORDINGS_BUCKET`)
- **Prefix**: `recordings/` (default, configurable via `SUPABASE_RECORDINGS_PREFIX`)
- **Max File Size**: 100 MB
- **Retention**: 48 hours (default, configurable via `SUPABASE_RECORDINGS_RETENTION_HOURS`)

**Location in code:** [server.ts, lines 101-110](server.ts#L101-L110)

### Recording Filename Format
Recordings are named with metadata embedded in the filename (since Supabase Storage doesn't support custom metadata):

```
{ISO_TIMESTAMP}__{PARTS}.{EXTENSION}
```

**Example:**
```
2024-05-15T14-30-45.123Z__consultation-12345__faculty-fac-001__student-student-uuid__number-2024001__facultyname-Dr.Alan.Turing__studentname-John.Doe.webm
```

**Parts (metadata encoded):**
- `consultation-{id}`: Queue/consultation ID from database
- `faculty-{id}`: Faculty ID
- `student-{id}`: Student UUID
- `number-{number}`: Student number
- `facultyname-{name}`: Faculty name (sanitized)
- `studentname-{name}`: Student full name (sanitized)

### Recording Metadata Extraction
**Location:** [server.ts, lines 1797-1814](server.ts#L1797-L1814)

The system can parse recording filenames to extract:
- `consultationId`
- `facultyId`
- `studentId`
- `studentNumber`

## 3. HOW RECORDINGS ARE LINKED TO CONSULTATIONS

### Link Mechanism: Filename Metadata + Database Query

**Step 1: Upload** ([server.ts, lines 6854-7015](server.ts#L6854-L7015))
- POST `/api/recordings/upload` accepts file + optional `consultation_id`, `faculty_id`, `student_id`
- If `consultation_id` is provided, the system queries the queue table to get faculty/student IDs
- All IDs are embedded in the filename for future querying

**Step 2: Retrieval** ([server.ts, lines 6629-6770](server.ts#L6629-L6770))
- GET `/api/recordings` lists all recordings from Supabase bucket
- Parses filenames to extract metadata
- Queries the queue table using extracted `consultationId` to get full consultation data
- Joins with faculty and students tables to get names

**Step 3: Metadata Flow**
```
Recording Upload
    ↓
Extract/store in filename: consultation_id, faculty_id, student_id, student_number
    ↓
Stored in Supabase Storage bucket
    ↓
Recording Retrieval
    ↓
Parse filename → Get consultation_id, faculty_id, student_id
    ↓
Query queue table WHERE id = consultation_id
    ↓
Join with faculty/students tables for display names
```

## 5. HOW TO QUERY RECORDINGS BY FACULTY_ID

### Method 1: Filter GET `/api/recordings` Response in Frontend
```javascript
// Get all recordings
const res = await fetch('/api/recordings?pageToken=0');
const data = await res.json();

// Filter by faculty_id
const facultyRecordings = data.files.filter(file => file.facultyId === 'fac-001');
```

### Method 2: Parse Filenames Directly (Backend)
```typescript
const recordings = await supabase.storage
  .from('consultation-recordings')
  .list('recordings');

const facultyRecordings = (recordings.data || [])
  .filter(file => {
    const parsed = parseSupabaseRecordingObjectName(file.name);
    return parsed.facultyId === 'fac-001';
  });
```

### Method 3: Query via Consultation ID
```typescript
// Get all consultations for a faculty
const { data: consultations } = await supabase
  .from('queue')
  .select('id')
  .eq('faculty_id', 'fac-001');

// Get recording IDs from filenames
const consultationIds = consultations.map(c => c.id);
const recordings = await supabase.storage
  .from('consultation-recordings')
  .list('recordings');

const facultyRecordings = recordings.data.filter(file => {
  const parsed = parseSupabaseRecordingObjectName(file.name);
  return consultationIds.includes(parsed.consultationId);
});
```

## 6. RECORDING METADATA STORED WITH RECORDINGS

### Metadata in Filename (Parsed by System)
1. **Upload Timestamp** - ISO timestamp in filename
2. **Consultation ID** - `consultation-{id}` segment
3. **Faculty ID** - `faculty-{id}` segment
4. **Student ID** - `student-{id}` segment
5. **Student Number** - `number-{studentNumber}` segment
6. **Faculty Name** - `facultyname-{name}` segment (sanitized)
7. **Student Name** - `studentname-{name}` segment (sanitized)

### Metadata Available from Related Database Queries
From **queue** table:
- `status`, `start_time`, `end_time`, `queue_date`, `time_period`, `purpose`
- `meeting_link` (Google Meet link if applicable)
- `recording_enabled` flag

From **faculty** table:
- `name`, `full_name`, `email`, `department_id`, `college_id`

From **students** table:
- `full_name`, `email`, `course`, `student_number`

### File Metadata Available from Supabase Storage
- `created_at` - Upload time
- `updated_at` - Last update time
- `size` - File size in bytes
- `mime_type` - Content type

## Summary for Faculty Dashboard Implementation

### Data Flow:
1. **Query** current faculty's consultations from `queue` table (WHERE `faculty_id = ?`)
2. **Fetch** recordings list from `GET /api/recordings`
3. **Filter** recordings by `facultyId` to get faculty's recordings
4. **Enrich** with consultation metadata (student name, date, purpose, status)
5. **Display** in UI with play/download options
6. **Stream** recording via `GET /api/recordings/content?path=...`

### Key Query: Get Faculty's Recordings with Full Context
```typescript
// 1. Get consultations
const { data: consultations } = await supabase
  .from('queue')
  .select(`
    id,
    student_id,
    students(full_name, student_number),
    start_time,
    end_time,
    status,
    purpose
  `)
  .eq('faculty_id', facultyId)
  .order('start_time', { ascending: false });

// 2. Get recordings
const recordingsRes = await fetch('/api/recordings');
const { files } = await recordingsRes.json();

// 3. Link recordings to consultations
const recordingsWithConsultation = files
  .filter(f => f.facultyId === facultyId)
  .map(recording => ({
    ...recording,
    consultation: consultations.find(c => c.id.toString() === recording.consultationId)
  }))
  .filter(r => r.consultation); // Only show recordings with matching consultation
```

This gives you complete context for each recording including student name, date, time, purpose, and current status.
